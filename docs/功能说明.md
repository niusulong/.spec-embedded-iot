# spec-knowledge-archiver 功能说明

> 嵌入式 IoT 知识库的**归档 + 向量索引 + 语义检索**工具。把项目 `.spec/` 下的分析与方案文档归档到跨项目持久化知识库，并构建可语义检索的向量索引。

## 一、定位

| 维度 | 说明 |
|------|------|
| 输入 | 项目 `.spec/{bug,requirement}/` 下的 markdown 文档（可多文件） |
| 输出 | 持久化知识库 `~/.spec-embedded-iot/knowledge/platform/{平台}/{类型}/`：`.archive_meta.json`（元数据）+ `index.md`（人类可读索引）+ `*.md`（归档正文）+ ChromaDB 向量索引 |
| 跨项目 | 知识库独立于任何代码仓库，按芯片平台隔离（EC626 / ASR1603 / N58 / UIS8850 / UIS8852 / QCX216 …） |
| 配置驱动 | 新增知识类型通常只需在 `knowledge_config.json` 加一条配置 |

## 二、数据流

```
归档链路                          向量链路                         检索链路
.spec/{bug,requirement}/    →   knowledge/platform/{平台}/{dest}/   →   embed_search.py
  Bug分析.md / 需求.md            .archive_meta.json (元数据)            跨 collection 语义检索
  方案.md / 任务.md (多文件)      index.md (人类可读表)                   按 (platform,file) 去重
       │                          *.md (归档正文)
  knowledge_archiver.py           │
  - 合并多文件(主文档优先)        embed_indexer.py
  - 注入 工作项ID/项目ID=NA       - summary 策略: 摘要字段向量
  - hash 增量跳过                   + index_body: 正文分块向量(混合)
  - 原子写 + meta/index 重建       - markdown_chunks 策略: 全文分块
                                   - ChromaDB PersistentClient (单库)
                                   - paraphrase-multilingual-MiniLM-L12-v2
```

归档后**自动**调用 `embed_indexer.py update` 更新向量索引（可用 `--no-vector` 跳过）。

## 三、脚本职责（`skills/spec-knowledge-archiver/scripts/`）

| 脚本 | 职责 | 关键函数 |
|------|------|----------|
| `knowledge_archiver.py` | 归档主程序：CLI（archive/status/verify）、多文件合并、hash 增量、字段注入、迁移、meta/index 重建 | `archive_entry`、`merge_md_files`、`ensure_summary_field_row`、`generate_index`、`cmd_verify` |
| `embed_indexer.py` | 向量索引构建/更新：summary 与 markdown_chunks 双策略、混合索引（summary+body）、孤儿清理 | `_collect_summary_entries`、`_collect_markdown_chunks`、`index_all`、`update_index` |
| `embed_search.py` | 语义检索：跨 collection、按平台过滤、按源文件去重、`[摘要]`/`[正文]` 标签、JSON 输出 | `_query_collection`、`search` |
| `extract_summary.py` | 从 markdown 提取结构化摘要（解析摘要节表格 → 全文兜底 → 首段兜底） | `_parse_table_rows`、`_find_summary_section`、`extract_summary` |
| `common.py` | 共享：路径常量、配置加载、向量客户端懒加载、路径/文件名工具、摘要文本构建、markdown 分块、原子 IO | `load_config`、`get_vector_client`、`build_summary_text`、`chunk_markdown`、`safe_filename` |

## 四、配置 Schema（`~/.spec-embedded-iot/knowledge/knowledge_config.json`）

配置分两部分：

### `doc_types`（归档脚本用）
```jsonc
"bug": {
  "source_dir": "bug",              // 项目 .spec/ 下的源目录
  "dest_dir": "bug-solutions",      // 知识库目标目录（== collection 名）
  "index_columns": { ... },         // index.md 表格列
  "summary_section": "结构化摘要",  // 摘要节标题
  "summary_fields": { ... },        // 摘要字段映射（中文 label → 英文 key）
  "list_fields": ["symptoms", "keywords"],
  "max_field_lengths": { "root_cause": 60 }
}
```

### `collections`（向量索引用）
```jsonc
"bug-solutions": {
  "strategy": "summary",            // 或 "markdown_chunks"
  "sources": ["platform/*/bug-solutions/.archive_meta.json"],
  "index_body": true,               // summary 策略专属：额外对正文分块建向量
  "chunk_size": 1200,
  "chunk_overlap": 200
}
```

**两种索引策略**：

| 策略 | 适用 | 向量来源 |
|------|------|----------|
| `summary` | 小文档+结构化摘要（bug/requirement） | `.archive_meta.json` 的摘要字段 |
| `markdown_chunks` | 长文档（code-summary/official-docs/protocols/project-overview） | `.md` 全文分块 |

### 混合索引（summary + `index_body:true`）

bug-solutions / requirement-solutions 默认开启。每个条目产出两类向量：

| 向量 | `doc_kind` | 来源 | 作用 |
|------|-----------|------|------|
| 摘要向量 | `summary` | 结构化摘要 8 字段 | 提精度 + 保结构化过滤 |
| 正文块向量 | `body` | `.md` 正文分块 | 保召回（日志/寄存器/AT 命令等细节可检索） |

> 这样无论文档有没有写结构化摘要表都能被检索到，摘要只是提升精度而非检索前置门槛。

## 五、命令清单

### 归档（`knowledge_archiver.py`）
```bash
python scripts/knowledge_archiver.py archive --project {项目路径} --type bug --all          # 批量
python scripts/knowledge_archiver.py archive --project {项目路径} --type bug --incremental   # 增量
python scripts/knowledge_archiver.py archive --project {项目路径} --type bug --name "COAP"   # 指定条目
python scripts/knowledge_archiver.py status  --project {项目路径}                            # 状态
python scripts/knowledge_archiver.py verify  --project {项目路径} --type bug                 # 完整性校验
python scripts/knowledge_archiver.py archive --project {项目路径} --platform EC626 --type bug --all  # 手动指定平台
```

### 向量索引（`embed_indexer.py`）
```bash
python scripts/embed_indexer.py build                       # 全量重建所有 collection
python scripts/embed_indexer.py build --collection bug-solutions
python scripts/embed_indexer.py update                      # 增量 upsert + 孤儿清理
python scripts/embed_indexer.py status
```

### 检索（`embed_search.py`）
```bash
python scripts/embed_search.py "MQTT QoS"                                    # 跨 collection
python scripts/embed_search.py "LWIP 内存泄漏" --collection bug-solutions     # 限定 collection
python scripts/embed_search.py "MQTT 连接失败" --collection bug-solutions --platform EC626  # 限定平台
python scripts/embed_search.py "死机" --top 10 --json                        # JSON 输出
```

检索输出用 `[摘要]`/`[正文]` 标签区分命中类型，`[正文]` 命中显示命中的段落标题；按源文件去重，同一文档多个块只保留最相关一条。

## 六、知识库目录布局

```
~/.spec-embedded-iot/knowledge/
├── knowledge_config.json                 [运行时配置]
├── platform/                             [源数据：归档原文 + meta]
│   └── {平台}/
│       ├── bug-solutions/                .archive_meta.json + index.md + *.md
│       ├── requirement-solutions/        同上
│       ├── code-summary/{模块}/代码总结.md
│       ├── official-docs-md/
│       └── 项目概览.md
├── protocols/                            [协议官方资料]
└── vector_db/                            [生成数据：ChromaDB，已 gitignore]
```

平台名从项目路径推断（`D:\EC626\` → `EC626`），也可 `--platform` 手动指定。

## 七、关键设计约定

1. **工作项ID/项目ID 追溯**：所有归档文档的摘要须含 工作项ID（bug）/ 项目ID（requirement），缺失时自动注入 `NA`。
2. **主文档置顶合并**：多文件归档时，含「结构化摘要」节的文档置顶，section 间用 `\n\n---\n\n` 连接。
3. **hash 增量**：基于源文件二进制算 hash，内容+文件名都未变则跳过；注入的 NA 行不进 hash（避免死循环重写）。
4. **原子写 + 写新再删旧**：迁移时先 atomic_write 新文件再删旧文件，防数据丢失。
5. **两个隐式约定**：`doc_type.dest_dir == collection 名`；summary 策略 collection 须能反查到 doc_type。（审查指出这是脆弱点，见审查报告 H4/Tier 2）
