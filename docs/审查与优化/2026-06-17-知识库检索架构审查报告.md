## 一、当前向量知识库架构

```
归档链路                          向量链路                         检索链路
.spec/{bug,requirement}/    →   knowledge/platform/{平台}/{dest}/   →   embed_search.py
  Bug分析.md / 需求.md            .archive_meta.json (元数据)            跨 collection 语义检索
  (多文件)                        index.md (人类可读表)                   按 (platform,file) 去重
       │                          *.md (归档正文)
  knowledge_archiver.py           │
  - 合并多文件(主文档优先)        embed_indexer.py
  - 注入 工作项ID/项目ID=NA       - summary 策略: 摘要字段向量
  - hash 增量跳过                   + index_body: 正文分块向量(混合)
  - 原子写 + meta/index            - markdown_chunks 策略: 全文分块
                                   - ChromaDB PersistentClient (单库)
                                   - paraphrase-multilingual-MiniLM-L12-v2
```

**两个隐式约定桥接三链路**：① `doc_types.{type}.dest_dir` 必须 == `collections` 的某个键；② summary 策略的 collection 必须能反查到 doc_type。整个系统配置驱动，设计意图清晰，规模（4 平台 / ~30 文档 / 15M 库）下完全胜任。

## 二、审查方法

4 个并行 subagent 分头审查（设计原则 / 归档准确性 / 检索与向量架构 / 目录配置可扩展性），对其中 5 条最高影响结论**逐一代码复核**，并修正了 1 条（迁移漂移：agent 判为"当前正在发生"，实测为"潜在风险"）。

### 🟡 次要 / 改进

- **M1** chunk_markdown 空输入返回 `[('', '')]` 而非 `[]`；单个超大段落（无空行）无法切，产出超 chunk_size 巨型块。
- **M2** 过取因子偏小（`top*5`），大文档可能独占配额饿死其它文件；去重每文件仅留 1 块（同一文档不同段落都可能相关）。
- **M3** 去重 key 未含 collection 维度（当前路径不碰撞，但脆弱）；doc_id 跨平台分隔符不一致（Windows 反斜杠）。
- **M4** `index_columns` 与 `summary_fields` 职责重叠、label 冲突（root_cause 在表格叫"根因方向"、摘要叫"根因概述"）。
- **M5** "结构化摘要"标题正则有 3 套不一致写法（merge/ensure/extract 各一）。
- **M6** 死代码：`get_collection()`、`scan_platforms()` 定义未调用。

## 四、优化方案（分 4 层优先级）

### Tier 0 — 立即修的正确性 bug（均已验证，工作量小）

| 优先 | 修复 | 做法 | 工作量 | 依赖 |
|------|------|------|--------|------|
| **C1** | verify 校验复活 | `COLLECTION_NAME` 改为 `from common import list_collections` 遍历，或定义常量；`except:pass` 改 `print(warn)` | S | — |
| **C2** | 改用 cosine 空间 | 3 处 `get_or_create_collection` 的 metadata 加 `{"hnsw:space":"cosine"}`；**必须 `build` 全量重建**（update 不改已有 space） | S + 一次 build | C1 后顺手 |
| **C3** | chunk_markdown 不切标题 | 大节分支用 `body.split("\n\n")` 而非 `section_text`；开头 `if not content.strip(): return []` | S | 重建索引 |
| **C4** | 表格正则严格化 | value 改 `([^|]*)` 不含竖线，或加 `row.count("\|")` 列数校验 | S | 重新归档受影响条目 |

> ⚠️ C2/C3 修复后需 `build` 重建全部 collection（cosine 改动 + 去垃圾块）。

### Tier 1 — 重要可靠性

| 优先 | 修复 | 做法 | 工作量 |
|------|------|------|--------|
| **H1** | 相似度阈值 | 加 `--min-similarity`（cosine 校准后默认 ~0.3），低于则丢弃并提示 | S |
| **H2** | verify 反向校验 | 遍历 chroma 全 id，拆出 `{plat}/{file}` 在 meta 找不到则报 `[僵尸向量]`；向量检查去 `doc_type=="bug"` 限制 | M |
| **H3** | 解耦向量 id 与文件名 | 向量 id 改用 `entry_name`（稳定），文件名只进 metadata；迁移不再产生 id 漂移 | M |
| **H5** | 库函数抛异常 | `get_*_config` 抛 `ValueError`，`sys.exit` 只留 `main()` | S |
| **H6** | 静默 except 降级为 warn | 至少保留 stderr 输出（重点 verify/delete_collection/extract --type 三处） | S |

### Tier 2 — 架构/可维护性

| 修复 | 做法 | 工作量 |
|------|------|--------|
| **H4** 配置 schema 校验 + 深度合并 | `load_config` 校验必需键、断言 summary collection 能反查 doc_type、与默认值深度合并、version 迁移函数 | M |
| doc_type↔collection 显式关联 | doc_type 配置加 `"collection"` 字段，消除 `dest_dir==collection_name` 隐式反查 | S |
| 抽 `SUMMARY_HEADING_RE` 常量 | 统一 3 处"结构化摘要"正则 | S |
| 缓存加锁 + deepcopy + `reload_config()` | `load_config` 返回前 deepcopy；ef 单例加 Lock | S |
| 去重 key 加 collection + 过取因子调大 | `key=(collection,platform,file)`；`fetch_n=max(top*10, top+50)` 或每文件留 top-K | S |

## 五、推荐实施顺序

1. **先 Tier 0 四条**（C1→C2→C3→C4），全部 S 工作量，消除已验证的正确性 bug；C2+C3 完成后**重建全部索引**。
2. **再 Tier 1**：H1（阈值，需 C2 cosine 校准后才能定阈值）→ H5/H6（让代码可被安全复用和运维）→ H3（id 解耦，根治迁移漂移）→ H2（verify 补全）。
3. **Tier 2/3 按需**：H4 配置校验优先（防静默瘫痪）；测试和依赖声明作为长期安全网。

**最高 ROI 三件套**：C1（verify 复活，S）、C2（cosine，S+rebuild）、C3（去垃圾块，S）—— 都是已验证 bug、工作量小、直接决定检索可信度。
