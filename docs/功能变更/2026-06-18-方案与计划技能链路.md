# 功能变更归档：方案与计划技能链路（2026-06-18，v1.3.0）

> 归档 spec-embedded-iot v1.3.0 核心改动：新增 `spec-solution-designer`（技术方案）与 `spec-implementation-planner`（实施计划），补全"需求→方案→计划"链路。含设计决策、与 superpowers 的关系、skill-creator 审查与 11 项优化。

## 一、背景：链路缺两环

`spec-requirement-generator`（产出 `需求.md`，聚焦"要什么"）与 `spec-requirement-splitter`（拆分需求）之后，缺把需求落地的两环：

| 缺口 | 产物 | 回答 |
|------|------|------|
| 技术方案 | `方案.md` | 怎么做（架构/任务/资源/协议栈兼容/风险） |
| 实施计划 | `计划.md` | 怎么落地（分期/任务/验收/排期） |

generator 已故意停在"要什么"（遇技术方案讨论即打断、指向"方案设计阶段"），故用新技能补，不改老技能定位。

## 二、链路

```
spec-requirement-generator (需求.md)
   └─ spec-requirement-splitter (拆分，可选)
        └─ spec-solution-designer        → 方案.md   (技术方案)
             └─ spec-implementation-planner → 计划.md (实施计划)
                  └─ [可选] superpowers: writing-plans / executing-plans
```

三份文档并排在 `.spec/requirement/{项目ID}_{核心功能}/`，路径约定与 generator 一致、带项目 ID 追溯。

## 三、两个技能

### spec-solution-designer（v1.3）
- **输入**：`需求.md`（缺失则转交 generator）。
- **流程**：读取需求 + 探索代码上下文 + 识别平台 → 探索式提问约束 → **提出 2-3 个备选方案 + 权衡 + 推荐** → 增量呈现关键章节逐段确认 → 生成 `方案.md` → **自审（占位符/一致性/范围/歧义/YAGNI + 校准）+ 用户审批 gate**。
- **复杂方案可选**：派独立 reviewer 子会话冷眼复查（`solution-reviewer-prompt.md`，嵌入式口味）。
- **配套**：`solution-template.md`（14 章，含「方案选型」）。

### spec-implementation-planner（v1.2）
- **输入**：`方案.md`（+`需求.md`；缺失则转交 solution-designer）。
- **流程**：读取方案 + 识别平台 → 提问实施约束 → **文件/模块结构映射** → 分期划分 → **反空话 WBS + 接口契约（单一事实源）** → 依赖关键路径 → 生成 `计划.md` → **自审（方案/需求覆盖/占位符/接口一致）+ 用户确认**。
- **核心机制**：反空话（No Placeholders，每任务必过）、接口契约表（防集成时名字对不上）。
- **可选转交**：`superpowers:writing-plans`（代码级展开）/ `executing-plans`，**仅当可用技能列表含该技能才提议**（可观测降级）。
- **配套**：`plan-template.md`、`anti-placeholder-examples.md`（反空话正反对照）、`plan-reviewer-prompt.md`。

## 四、关键设计决策

1. **拆两个技能而非一个**：仿 superpowers 的 brainstorming/writing-plans/executing-plans 模块化；方案与计划职责不同，分技能更清晰。
2. **同目录并排**：需求/方案/计划同处 `.spec/requirement/{项目ID}_{核心功能}/`，便于追溯与对照。
3. **与 superpowers 的关系——借鉴，不替代/包装**：
   - superpowers 的 brainstorming ≈ solution-designer、writing-plans ≈ implementation-planner，但前者**通用**，缺嵌入式维度（RTOS 任务/内存/功耗/协议栈兼容）与本项目的路径/模板/项目 ID 约定。
   - 不包装：brainstorming 有硬编码终态（必须调 writing-plans）与自己的输出路径，包装会控制流打架。
   - 借鉴其方法论（反空话、校准式自审、独立子会话复查），spec 保持自包含；superpowers 仅作计划技能末尾**可选转交**，并优雅降级。
4. **高度澄清**：本 `计划.md` 是**项目级**计划（分期/WBS/验收/排期）；superpowers `writing-plans` 是**代码级**计划（精确文件/代码/命令，给 agent 机械执行）。两者互补不同高度，不互相替代。

## 五、skill-creator 审查与优化（2026-06-18）

用 skill-creator 方法论审查（独立 reviewer 子会话 + 作者交叉核对），落地 11 项优化：

| 严重度 | 项 | 说明 |
|---|---|---|
| 高 | H1 | 触发词按**前置文件**锚定（需求.md vs 方案.md），description 前置文件存在性，避免两技能触发混淆 |
| 高 | H3 | superpowers 去三处重复 + **可观测降级**（看可用技能列表） |
| 中 | M1 | 新建 `anti-placeholder-examples.md`（反空话正反对照） |
| 中 | M2 | 方案选型 B/C 标可选，防硬凑稻草人 |
| 中 | M3 | solution Step 4 加跳过条件（小改动免四层 gating） |
| 中 | M4 | reviewer 派发改**可观测阈值**（≥2 新任务/动协议栈/RAM>10%） |
| 中 | M5 | 跨文档引用改**按章节名**（去硬编码 `## 13`） |
| 低 | L1–L5 | reviewer 理由去重 / description 精简 / 所属分期后填 / Step 1 平台识别 |

**纠正误报 1 项**：审查 H2"不实承诺 KB 归档"经核实为误报——技能用中性措辞，未承诺检索；真实残留（archiver 无 solution/plan doc_type）属已暂不考虑的 KB 归档范畴。
