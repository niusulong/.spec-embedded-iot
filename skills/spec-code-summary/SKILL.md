## 核心定位

这是单个模块的"实现说明书"：读一个模块的代码，讲清楚它**内部怎么实现的**——数据结构、接口、核心流程、状态机、错误处理，以及它在运行期日志里会留下哪些**可供检索的关键字**。让用户和后续的 spec 技能（尤其是 bug 分析）不必再逐行读代码就能掌握这个模块。

五个原则：
1. **单模块深入**：每次只分析一个模块的内部实现细节，函数级粒度（区别于 spec-project-overview 的目录级全景）。
2. **实现导向**：聚焦代码层面的实现机制（数据怎么流、状态怎么转、错误怎么处理），而非架构组织罗列。
3. **业务场景符合性**：梳理实现时对照业务场景核验——这套实现是否真的服务了该模块的业务目标？有没有"代码写了但业务没用上"或"业务需要但代码缺失"的偏差？把疑点标出来。具体可问自己：① 对外承诺的功能代码是否都实现了？② 有没有被定义但无任何调用方使用的接口/全局量（无效遗留）？③ 有没有 demo/测试性质、参数硬编码、失败不重连、数据不落地等不适合直接产品化的实现？发现的偏差在 §7 或附录标出。
4. **推断要给依据**：每个结论附出处（哪个文件/函数/行号），不编造。尤其日志检索字段，必须从代码里真实的日志调用提取字面量。
5. **依赖引用**：新模块依赖已有总结时，引用避免重复，只写本模块差异部分。

## 执行流程（严格按步骤）

### Step 1：确定分析范围

从用户输入直接提取：
- **模块名称**：用于文件命名和文档标题（如"AT命令模块"、"MQTT模块"）。
- **代码路径**：待分析的代码目录（相对或绝对路径）。

未给全则先问清，不要猜。

模块命名建议：

| 模块类型 | 建议命名 |
|---------|---------|
| AT 命令 | `AT命令模块` |
| 协议栈 | `{协议名}模块`（如 MQTT模块、LwM2M模块） |
| 驱动 | `{外设名}驱动`（如 UART驱动） |
| 业务逻辑 | `{功能名}模块`（如 数据采集模块） |

### Step 2：环境检查 + 确定平台

1. 检查 `.spec` 是否存在；不存在则提示先跑 `spec-init`，但分析本身不依赖它，可继续。
2. **平台名**（如 EC626、ASR1603、UIS8850）：
   - 优先从用户输入获取；
   - 或从项目路径推断（如 `D:\EC626\` → `EC626`）。
   > 注意：平台名只代表芯片系列，**不代表 OS**。OS 看 kernel 目录的实际源码（freertos 的 tasks.c / ThreadX 的 tx_ 前缀），不要凭平台名猜。
3. 知识库输出路径：`~/.agents/knowledge/platform/{平台名}/code-summary/{模块名}/代码总结.md`
   （Windows 下 `~` 展开为 `C:\Users\<用户>\`）。

### Step 3：代码分析与文档生成

**可用工具**（注意：本环境**没有** `list_directory` 工具）：

| 工具 | 用途 | 示例 |
|------|------|------|
| `Glob` | 按模式查找文件 | `Glob(pattern="**/*.h", path=<模块路径>)` |
| `Grep` | 搜索代码内容 | `Grep(pattern="typedef struct", path=<模块路径>)` |
| `Read` | 读取文件全文 | `Read(file_path=<绝对路径>)` |
| `Bash` | 列目录、统计 | `ls <路径>`、`find <路径> -name "*.c"` |

> 工具按需用，不为走流程而走流程：用户已给出明确文件清单或范围很小时，可跳过 Glob 直接 Read。

**文件解析优先级**（先抓骨架再填细节）：

| 优先级 | 文件类型 | 说明 |
|--------|----------|------|
| P0 | `*_types.h`、`*_api.h` | 核心类型和对外接口 |
| P1 | `*_core.c`、`*_main.c` | 核心实现 |
| P2 | `*_config.h` | 配置定义 |
| P3 | `*_port.c`、`*_hal.c` | 硬件适配 |

**分析内容**（对应模板章节）：
1. 目录结构 → 文件组织、关键文件及优先级（→ §3 代码结构）
2. 数据结构 / 接口 → 结构体、枚举、API、回调（→ §5、§6）
3. 实现机制 → 核心流程、状态机、错误处理（→ §7）
4. **日志检索字段** → 从代码里真实的 `LOG_*`/`printf`/`NWY_APP_LOG_*`/`OSI_LOGI` 调用提取可供 grep 的字面量：模块 TAG、任务名、错误码、状态字符串、URC、malloc/free 入口（→ §8）
5. 模块依赖 → 确定引用内容，避免与已有总结重复（→ §4）

**第三方库边界**：很多模块是"对第三方原始库的封装/适配"（如 libcoap、mbedtls、lwip、littlefs）。分析聚焦**本项目的封装/适配层**；被封装的第三方原始库只在 §4 模块依赖关系里提及其名称与版本，不深入分析其内部实现。

### Step 4：生成结构化文档

- **输出路径**：`~/.agents/knowledge/platform/{平台名}/code-summary/{模块名}/代码总结.md`
- **使用模板**：读 `references/code-summary-template.md`，填占位符后输出。
- §8「关键日志检索字段」的字段值必须是**代码里真实出现的字面量**（含大小写、`0x` 前缀、`CME ERROR:` 的冒号空格），每条带出处（函数:行）——这是给 spec-bug-analyzer / spec-memory-leak-analyzer 直接 grep 用的，描述或编造都没价值。

### Step 5：完整性检查 + 归档衔接

- 模板章节齐全；识别不到的章节明确写"未识别到，原因…"，**不留空更不编造**（§8 若模块无日志输出要说明）。
- Mermaid 代码块闭合、表格格式正确、引用链接有效。
- §8 字段值可 grep：抽查 2-3 个关键字能否用 `Grep` 在源码里命中。
- 生成后提示：可运行 `spec-knowledge-archiver` 纳入向量索引，供 bug 分析 / 方案设计检索复用。

## 模块依赖识别方法

| 依赖类型 | 识别方法 | 示例 |
|----------|----------|------|
| 头文件依赖 | 查 `#include` | `#include "at_api.h"` |
| 函数调用依赖 | 查外部函数调用 | `at_register_cmd()` |
| 数据结构依赖 | 查外部结构体 | `at_context_t` |
| 宏定义依赖 | 查外部宏 | `AT_MAX_CMD_LEN` |

## 扩展点识别方法（可选，有则记入附录 B）

| 扩展类型 | 识别方法 |
|----------|----------|
| 回调函数 | 查 callback_t、register |
| 配置表 | 查 const 数组、config 表 |
| 虚函数表 | 查函数指针结构体 |
| Weak 函数 | 查 `__attribute__((weak))` |

## 加密文件提示

若 `Read` 返回乱码或读不到内容，源码可能被 EsafeNet（绿盾）加密——转用 `esafenet-file-io` 技能透明读写（仅 Windows）。

## 输出格式要求

1. **必须有目录**：开头 `## 目录` + 锚点链接。
2. **优先 Mermaid**：架构图 / 流程图 / 状态图用 Mermaid。
3. **表格化**：数据结构、接口、配置、日志字段用表格。
4. **引用链接**：依赖模块用相对路径引用。
5. **日志字段用字面量 + 出处**：§8 的核心规矩。

## 常见问题

**Q：代码量太大如何分析？**
A：按 P0→P1→P2→P3 优先级解析，先画架构 / 流程骨架再填细节；非核心文件只在 §3 代码结构里列名，不展开。

**Q：新模块依赖已有总结，如何避免重复？**
A：§4 模块依赖关系里引用已有总结，正文只写本模块特有实现。
