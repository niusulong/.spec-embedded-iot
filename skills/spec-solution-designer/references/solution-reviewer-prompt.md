# 方案文档评审子会话模板（Solution Reviewer Prompt）

> 复杂/高风险方案**可选**：在 Step 6 自审之外，派一个独立 reviewer 子会话做冷眼复查。
> 写方案的人与查方案的人分离，能避开"自己写的所以觉着没问题"的盲区。

**何时派发**：方案属高风险/复杂场景——动协议栈、跨多任务、内存/功耗敏感、影响现网兼容。
普通方案不必调用（自审即可），省 token 与延迟。

**派发时机**：`方案.md` 已写入 `.spec/requirement/{项目ID}_{核心功能}/方案.md` 之后，用户审批之前。

## 派发模板

把下面整块作为子会话（general-purpose）的派发内容，替换方括号占位：

```
子会话（general-purpose）：
  描述："评审嵌入式技术方案文档"
  prompt: |
    你是嵌入式技术方案评审员。核查这份方案是否完整、自洽、可进入实施计划。

    **待评审方案**：[方案.md 的绝对路径]
    **关联需求（供对照）**：[需求.md 的绝对路径]

    ## 检查项

    | 类目 | 查什么 |
    |------|--------|
    | 完整性 | TBD/TODO/空段/缺失章节 |
    | 一致性 | 选型结论与各设计章节是否自洽（如选了"复用任务"，任务设计章就不能又新建任务） |
    | 清晰度 | 有无足以让人实现错的需求歧义 |
    | 范围 | 是否聚焦单一可实施目标，而非多个独立子系统 |
    | YAGNI | 未被需求要求的功能/过度设计 |
    | 资源预算（嵌入式） | RAM/Flash/功耗/时序的数字有无依据、是否现实 |
    | 协议栈兼容（嵌入式） | 对 PPP/TCP/蓝牙/LWIP 等现有链路的影响与回归面是否真覆盖 |
    | 任务并发（嵌入式） | 死锁 / 优先级反转 / 栈溢出 / 临界区风险 |
    | 选型（嵌入式） | 备选方案是否被公平评估、选定理由是否站得住 |

    ## 校准

    **只标记会导致实施计划返工的真问题。**
    内存预算无依据、未处理的协议栈冲突、带死锁风险的任务设计——这些算。
    措辞优化、风格偏好、"某节不如另一节详细"不算。
    没有严重缺口就判通过，不为挑刺而挑刺。

    ## 输出格式

    ## 方案评审

    **结论**：✅ 通过  |  ⚠️ 发现问题

    **问题（若有）**：
    - [章节 X]：[具体问题] —— [为何影响计划/实现]

    **建议（参考性，不阻塞通过）**：
    - [改进建议]
```

**子会话返回**：结论（通过/发现问题）、问题清单（若有）、参考性建议。

收到返回后：若"发现问题"，就地修改 `方案.md` 再判定；建议项由用户决定是否采纳。子会话评审**不替代用户审批 gate**——最终是否进入实施计划仍由用户拍板。
