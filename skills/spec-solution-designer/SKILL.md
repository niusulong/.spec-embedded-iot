## 核心原则
1. **聚焦"怎么做"**：基于已有需求给出技术实现方案，不重新讨论"要什么"
2. **备选先行**：定稿前先提 2-3 个可行方案 + 权衡 + 推荐——选型本身就是方案的核心价值，直接给单一答案等于跳过最关键的一步
3. **增量验证 + 审批 gating**：关键章节边呈现边确认；方案须用户批准才交接实施计划，基于未确认假设排期是返工的最大来源
4. **嵌入式思维**：始终覆盖任务/ISR、内存、功耗、时序、协议栈兼容这些嵌入式特有的实现维度
5. **结构化输出**：严格按 `references/solution-template.md` 生成 `方案.md`
6. **命名合规**：对外接口/函数/AT 命令/宏控的命名遵循 `spec-neoway-coding-standards`——在设计阶段就定下合规命名，避免到实施阶段返工

## 执行流程

### Step 1：读取需求并探索相关上下文
1. 在 `.spec/requirement/{项目ID}_{核心功能}/` 下查找 `需求.md`；读不到则询问位置；**完全没有** → 引导先运行 `spec-requirement-generator`
2. 读取需求，提取项目 ID、模块、核心功能、非功能约束（实时性/功耗/内存）
3. **探索相关现有代码与调用链**：定位涉及的模块/文件、现有任务、可复用接口与配置；**尤其核查共享/变体文件（命令表/配置/Kconfig/CMake/驱动适配）对本项目/变体的实际适用范围，把证据写入现状分析**——避免误改不影响本项目的文件。嵌入式方案不能脱离现有代码结构空谈，现状分析越扎实选型越靠谱
4. **识别当前平台**：从项目路径推断（如 `D:\EC626\` → EC626）或询问。任务模型、协议栈、资源预算的讨论须贴合该平台——ASR ThreadX 与 EC FreeRTOS 的任务/PPP/TCP 栈差异很大，方案不能脱离平台空谈

### Step 2：逐个提问澄清实现约束
**提问原则**：一次只问一个问题，优先多选题；从约束不清楚处切入；已明确的维度直接跳过。

**提问维度**（按需探索，不必全部问完）：

| 维度 | 典型问题 | 目的 |
|------|----------|------|
| 分层落点 | "这个功能改在哪一层？AT/协议栈/应用/驱动？" | 定位改动范围 |
| 任务模型 | "需要新建 RTOS 任务吗？还是复用现有任务？优先级/栈大小？" | 并发设计 |
| 中断与同步 | "会涉及中断或任务间通信吗？队列/事件/信号量？" | 实时性与互斥 |
| 内存预算 | "RAM/Flash 增量有上限吗？" | 资源约束 |
| 功耗时序 | "对功耗或关键路径时序有要求吗？" | 非功能约束 |
| 硬件依赖 | "用到哪些外设/引脚/总线/时钟/电源？" | 硬件相关改动 |
| 协议栈兼容 | "会影响现有 PPP/TCP/蓝牙/LWIP 等链路吗？" | 兼容性与回归面 |
| 配置可扩展 | "需要宏控/NV/运行时配置吗？后续会扩展吗？" | 可维护性 |
| 宏控与生效边界 | "新功能用什么功能宏？要进哪些项目/客户版本？" | 规范命名 + 编译可见范围 |
| 降级回滚 | "失败时怎么降级？能回滚吗？" | 可靠性 |

**停止条件**：架构落点、任务模型、关键约束（内存/功耗/协议栈兼容）、新功能宏控与生效边界都已明确，或用户表示"就这些了"。

**宏控命名与生效边界（新功能必做）**
新增功能须按 Neoway 编码规范的宏控规则命名功能宏、并界定生效范围。先调 `spec-neoway-coding-standards` 查「宏控规范」对齐规则，再：
1. **拟候选功能宏名**：按功能宏格式 `FEATURE_NWY_[AT|OPEN]_FUNC_[FUNC1]_[BZ]` 给 2-3 个候选——`AT`/`OPEN` 区分版本(不区分可省略)、`FUNC` 大功能、`FUNC1` 子功能(可选)、`BZ` 行业特性(可选)；宏名**大写+下划线**且**不得出现项目名称**（如 `FEATURE_NWY_AT_FTP`、`FEATURE_NWY_SENSOR_LTR559`、`FEATURE_NWY_AT_AUDIO_DTMF`）。
2. **给推荐 + 理由**：从候选中推荐一个，说明取舍（粒度粗细、是否需要子功能/行业分支）。
3. **定生效边界（在哪些项目生效）**：明确宏在哪些项目/产品/客户版本打开——
   - 通用功能进基线，由项目宏 `NWY_PROJECT_*` 范围内的构建配置打开；
   - 客户定制用 `FEATURE_NWY_CUS_[INDEX]` 包裹（CUS 由软件经理制定，如 `FEATURE_NWY_CUS_MT`）；
   - 标注默认开/关，并保证**回滚性**（去掉宏之后编译须正常）。

选定的宏名与生效范围写入 `方案.md`「§9.2 配置项与宏控」。

### Step 3：提出 2-3 个备选方案并选型【核心步骤】
基于 Step 1-2，给出 **2-3 个可行方案**。差异点通常落在：分层落点 / 新建任务 vs 复用 / 同步 vs 异步 / 协议栈改法 / 配置方式。每个方案给出：
- 一句话思路
- 关键权衡（资源占用 / 实时性 / 兼容性 / 改动量 / 风险）
- 适用与不适用场景

然后给出**推荐方案 + 理由**，请用户确认或调整。

> 即使看似只有一个显而易见的方案，也要简述"为什么不选别的"（如 X 改动太大、Y 破坏协议栈兼容）——留痕便于后续追溯与评审。这一步的产出写进 `方案.md` 的「方案选型」章节。

### Step 4：增量呈现设计，关键章节逐段确认
就**推荐方案**逐段呈现设计，每段呈现后请用户确认，有问题立即回头改，而非憋到最后一次性甩出：

- 分层落点与模块划分
- 任务与并发设计（优先级 / 栈 / 同步互斥）
- 资源预算（内存 / 功耗 / 时序）
- 协议栈兼容与回归面

其余章节（接口 / 异常降级 / 风险 / 验证要点）可在 Step 5 写文档时一并补全。

**跳过条件**：若方案改动面小（单模块、不动协议栈/任务模型）且 Step 2 约束已充分明确，可省略逐段确认，直接进 Step 5 写文档 + Step 6 一次性审批——避免简单方案也走四道 gate。

### Step 5：生成技术方案文档
1. 读取 `references/solution-template.md`
2. 填充：方案选型（Step 3）、各设计章节（Step 4）、接口设计、异常降级、风险、验证要点
3. 关键流程用 Mermaid 画时序/状态机，模块关系用 Mermaid 画数据流
4. 输出到 `.spec/requirement/{项目ID}_{核心功能}/方案.md`
5. **必填「## 0. 结构化摘要」**（项目 ID / 模块 / 方案概述 / 影响面 / 关联需求文档）

### Step 6：方案自审 + 用户审批
**自审（写完用新眼光扫一遍，发现问题就地修）**：

| 检查项 | 查什么 |
|--------|--------|
| 占位符 | 有无 TBD/TODO/空段 |
| 一致性 | 选型结论与各章节是否自洽（如选了"复用任务"，任务设计章就不能又新建任务） |
| 范围 | 是否聚焦单一可实施目标；过大则建议先 `spec-requirement-splitter` 拆分 |
| 歧义 | 有无需求可被两种解读，有则定一种并写明 |
| YAGNI | 有无未被需求要求的功能/过度设计——嵌入式资源紧张，多建一个任务、多开一个缓冲区都是真金白银 |

**校准（重要）**：只标记**会导致实施计划返工的真问题**——遗漏的章节、自相矛盾、足以被两种解读的需求。措辞优化、风格偏好、"某节不如另一节详细"不算问题。没有严重缺口就判通过，不为挑刺而挑刺。

**裁决**：给出明确结论——`✅ 通过` 或 `⚠️ 发现问题（列出）`；问题就地修完后重新判定。

**复杂方案可选：独立子会话评审**
满足以下任一**可观测**条件时，自审之外派一个独立 reviewer 子会话冷眼复查（用 `references/solution-reviewer-prompt.md`）：方案涉及 ≥2 个新建任务、或改动协议栈/中断层、或预估 RAM 增量 > 现有 10%、或影响现网兼容。理由与派发模板见该 reference。普通方案自审即可，省 token 与延迟。

**审批 gate**：自审（必要时含子会话评审）通过后，请用户审阅 `方案.md` 并明确批准。**未批准不进入实施计划**——这一步挡掉绝大多数"基于想当然的假设排期"导致的返工。

## 交互规则
- 遇到"要什么"的功能/场景讨论 → 提醒："这是方案阶段，需求范围请回 spec-requirement-generator 确认"
- 方案经用户批准 → 提示下一步运行 `spec 实施计划`（spec-implementation-planner）把方案转成开发任务与排期
- 需求文档缺失 → 引导先 `spec 整理需求`（spec-requirement-generator）
- 方案范围过大 → 引导先 `spec 拆分需求`（spec-requirement-splitter）
