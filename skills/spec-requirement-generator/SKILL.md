## 核心原则
1. **聚焦"要什么"**：只描述功能、用户、场景，不涉及技术实现方案
2. **结构化输出**：严格按照标准模板生成.md文档
3. **探索式提问**：一次一个问题，优先多选题，逐步澄清需求
4. **嵌入式软件思维**：特别关注实时性、可靠性、功耗等嵌入式软件特性

## 执行流程

### Step 1：探索项目上下文
1. 检查 `.spec/requirement/` 目录下已有需求文档，了解当前项目状态
2. 识别与需求相关的已有代码模块或文档

### Step 2：逐个提问澄清需求

**提问原则**：
- **一次只问一个问题**，优先使用多选题降低回答成本
- **从宏观到微观**：先问背景目标，再问具体功能
- **需求清晰即停止**，不要过度提问

**提问维度**（按顺序探索，已明确的可跳过）：

| 维度 | 典型问题 | 目的 |
|------|----------|------|
| 背景 | "这个功能解决什么问题？" | 理解动机 |
| 用户 | "目标用户是谁？使用场景是什么？" | 明确受众 |
| 核心功能 | "最关键的功能有哪些？（可多选）" | 确定范围 |
| 优先级 | "这些功能中哪个是必须的？" | 区分P0/P1/P2 |
| 嵌入式约束 | "对实时性/功耗/内存有特殊要求吗？" | 捕获非功能需求 |
| 边界 | "哪些情况明确不需要支持？" | 排除范围 |

**停止条件**：用户已提供完整信息，或核心功能/优先级/边界都已明确，或用户表示"就这些了"。

### Step 3：确认核心功能、项目 ID 与文件名
1. 提取核心功能关键词（动词+名词，如"蓝牙配对"、"数据采集"）
2. 文档统一命名为 `需求.md`，置于 `.spec/requirement/{项目ID}_{核心功能}/` 下（目录名 3-8 字，避免泛称）

**确认项目 ID**：如果用户未提供项目 ID（如 6974423486），必须询问用户获取。项目 ID 是归档追溯的唯一来源，归档时写入知识库 index、元数据与检索结果。目录命名格式：`{项目ID}_{核心功能}`（如 `6974423486_MQTT_SSL双向认证`）。

### Step 4：生成结构化文档

1. 读取 `references/requirement-template.md`
2. 创建功能目录并填充模板后输出到 `.spec/requirement/{项目ID}_{核心功能}/需求.md`
3. **必填「## 0. 结构化摘要」**（项目 ID / 模块 / 需求描述 / 优先级）——此摘要是知识库检索的数据来源，归档时由脚本自动提取；缺失则无法归档追溯

### Step 5：完整性检查与交付
1. 生成完整文档后，主动展示给用户
2. 确认：结构是否符合预期、待澄清事项是否准确、是否有遗漏

## 交互规则
- 遇到技术方案讨论 → 提醒："这是需求阶段，建议在方案设计中讨论实现细节"，并提示后续可用 `spec 设计方案`（spec-solution-designer）产出 `方案.md`
- 文档生成完成 → 提示下一步：大需求可运行 `spec 拆分需求`（spec-requirement-splitter）拆分；准备做技术方案时运行 `spec 设计方案`（spec-solution-designer）
