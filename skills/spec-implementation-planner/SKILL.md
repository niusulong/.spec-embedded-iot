## 核心原则
1. **聚焦"怎么落地"**：基于已定的技术方案产出分期 + 任务 + 验收，不重新做技术选型
2. **反空话（No Placeholders）**：每条任务必须具体到"可验证产物 + 验证方式"。禁止"实现XX逻辑"、"处理异常"、"完善相关功能"、"与现有模块对接"、"类似任务N"这类没有验收判据的空话——这是计划能否被准确执行的关键
3. **验收驱动 + 嵌入式验证**：每个任务配验收点；验证方式贴合嵌入式实际（单测/AT命令测试/抓包/长时间运行/功耗测试），不强求 TDD 跑红跑绿
4. **任务可独立验证**：粒度取"带自己验证周期的最小单元"；setup/配置/脚手架并入需要它的任务，只在"评审者能单独否决此项而通过相邻项"处切分
5. **接口契约显式**：任务间产/消的接口名、签名要写明并保持一致，避免集成时名字对不上
6. **可执行优先**：自带模板保证产出可用，superpowers 作为可选增强而非依赖
7. **编码规范合规**：所有任务产出的代码须符合 Neoway 编码规范（`spec-neoway-coding-standards`：命名/注释/风格/宏控）——作为计划的 Global Constraint 写进 `计划.md`，使人工或 agent 执行都遵循

## 执行流程

### Step 1：定位并读取方案文档
1. 在 `.spec/requirement/{项目ID}_{核心功能}/` 下查找 `方案.md`，必要时同读 `需求.md`
2. 用户已给路径则直接读；未找到则询问；**完全没有方案文档** → 引导先运行 `spec-solution-designer`
3. 读取方案的：分层落点、任务模型、资源预算、**验证要点**、风险
4. **识别当前平台**：从项目路径推断或询问。任务模型与验证方式（ThreadX vs FreeRTOS、AT/抓包手段）须贴合该平台

### Step 2：逐个提问澄清实施约束
**提问原则**：一次一个问题，优先多选题；约束已明确的跳过。

| 维度 | 典型问题 | 目的 |
|------|----------|------|
| 工期人力 | "大概工期多久？几个人投入？" | 排期与并行度 |
| 开发方式 | "哪些任务能写单测？哪些只能上板验证？" | 验证方式务实 |
| 合入策略 | "能否分批合入？需要灰度吗？" | 分期与回滚 |
| 回归范围 | "回归覆盖哪些？有自动化吗？" | 验证方式 |
| 执行引擎 | "要不要接入 superpowers 自动执行部分任务？" | 决定是否转交 |

**停止条件**：工期/人力、合入策略、验证手段已基本明确，或用户表示"就这些了"。

### Step 3：文件/模块结构映射【先于任务】
拆任务前先锁定代码分解——决定创建/修改哪些文件与模块、各自职责。这是后续分期与任务的骨架，也避免任务边界乱划。

| 文件/模块 | 操作 | 职责 | 所属分期（Step 4 后回填） |
|-----------|------|------|----------|
| [路径/模块名] | 新建/修改 | [单一明确职责] | P? |

遵循方案的分层落点；遵循现有代码结构，不擅自重构无关代码。

### Step 4：分期划分（Phase）
按"依赖顺序 + 可验证性"切分，每期必须有一个**可验证产物**。典型参考：

| 期号 | 目标 | 可验证产物 |
|------|------|-----------|
| P0 | 骨架可编译 | 接口/任务/配置就位，编译通过 |
| P1 | 核心通路 | 主流程端到端跑通 |
| P2 | 异常与降级 | 失败/边界场景按方案处理 |
| P3 | 优化与回归 | 资源预算达标、回归通过 |

展示分期建议，用户可调整。

### Step 5：任务拆解（WBS）+ 接口契约
把每期拆成任务卡片，每张卡填写：

| 字段 | 说明 |
|------|------|
| 任务 ID | T-P?-NN，如 T-P1-03 |
| 标题 | 动词开头的短描述 |
| 依赖 | 前置任务 ID |
| 预估 | 人时/人天 |
| 验收点 | **具体的、可勾选的完成判定** |
| 验证方式 | 单测/AT命令测试/抓包/长时间运行/功耗测试/编译 |
| 涉及文件 | 对应 Step 3 的文件/模块 |
| 产出接口 | 本任务对外提供的接口名/签名（供下游消费） |
| 消费接口 | 本任务依赖的上游接口名/签名 |

**反空话校验（每条任务必过）**：标题与验收点不得出现"实现XX逻辑/处理异常/完善XX/与XX对接/类似任务N"等没有可验证判据的表述——注意"变体空话"（如"实现 PPP 重拨逻辑"看似具体实则无验收判据）。拿不准就继续拆细或写具体验收，宁可多拆不可含糊。正反对照见 `references/anti-placeholder-examples.md`。

同时维护一张**接口契约表**（单一事实源，防集成时名字对不上）：

| 接口名 | 签名 | 产出任务 | 消费任务 |
|--------|------|----------|----------|
| [如 ppp_redial_start] | [原型] | T-P1-02 | T-P1-03, T-P2-01 |

### Step 6：依赖与关键路径
用 Mermaid 画任务依赖图，标出关键路径（决定总工期的最长依赖链）与可并行任务组。

### Step 7：生成实施计划文档
1. 读取 `references/plan-template.md`
2. 填充：概述、文件/模块结构映射、分期、WBS + 接口契约、依赖关键路径、验收验证矩阵、风险、合入回滚
3. 验收点尽量追溯到方案的『验证要点』章节（按章节名引用，不依赖章节号）
4. 输出到 `.spec/requirement/{项目ID}_{核心功能}/计划.md`（与 `方案.md`、`需求.md` 同目录）
5. **必填「## 0. 结构化摘要」**

### Step 8：计划自审 + 用户确认
**自审（写完用新眼光扫一遍，就地修）**：
- **方案/需求覆盖**：方案的验证要点、需求的功能项，能否各指到一个任务？列出缺口并补任务
- **占位符扫描**：有无空话任务（反空话校验）、TBD/TODO、没写验证方式的任务
- **接口一致性**：产出/消费的接口名在接口契约表与各任务间是否一致（`redial_start()` vs `ppp_redial()` 这类不一致就是 bug）

**校准**：只标会导致执行返工或集成对不上的真问题；没有严重缺口就判通过，不为挑刺而挑刺。

**裁决**：`✅ 通过` 或 `⚠️ 发现问题（列出）`；问题就地修完后重新判定。

**复杂计划可选：独立子会话评审**
满足以下任一**可观测**条件时，自审之外派一个独立 reviewer 子会话冷眼复查（用 `references/plan-reviewer-prompt.md`）：计划涉及 ≥2 个新建任务、或改动协议栈/中断层、或跨多模块集成、或影响现网兼容。理由与派发模板见该 reference。普通计划自审即可，省 token 与延迟。

**用户确认**：自审（必要时含子会话评审）通过后，请用户确认 `计划.md`。

## 接入 superpowers（可选，注意高度）
本技能产出的 `计划.md` 是**项目级计划**（分期/WBS/验收/排期）；superpowers 的 `writing-plans` 是**代码级计划**（精确文件/完整代码/精确命令+预期输出，给 agent 机械执行）。两者不同高度、互补。

**可观测判定（重要）**：仅当**当前可用技能列表里出现 `superpowers:writing-plans`** 时，才向用户提议转交；列表里没有就**完全不提** superpowers——`计划.md` 本身已是完整可用的项目级交付物。这把"优雅降级"变成一个可观测判断，而非模糊口号。

转交时：对适合 agent 执行的任务（通常应用层、可隔离测试的部分），调 `superpowers:writing-plans` 展开、`superpowers:executing-plans` 执行。嵌入式很多任务靠上板/AT/抓包验证，不适合 writing-plans 的纯 TDD 步进——挑合适的任务转交，别一刀切。**转交时明确要求**：所有生成的代码须符合 `spec-neoway-coding-standards`（命名/注释/风格/宏控）。

## 交互规则
- 遇到技术选型/架构争议 → 引导："这是计划阶段，技术决策请回 spec-solution-designer 确认"
- 计划确认 → 提示：可直接按计划推进；适合自动执行的部分可选转交 superpowers:writing-plans/executing-plans
- 方案文档缺失 → 引导先 `spec 设计方案`（spec-solution-designer）
