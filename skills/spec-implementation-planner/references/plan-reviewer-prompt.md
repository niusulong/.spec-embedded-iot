# 计划文档评审子会话模板（Plan Reviewer Prompt）

> 复杂/高风险计划**可选**：在 Step 8 自审之外，派一个独立 reviewer 子会话做冷眼复查。
> 写计划的人与查计划的人分离，能避开"自己写的所以觉着没问题"的盲区。

**何时派发**：计划跨多任务/多模块、或动协议栈等高风险场景。
普通计划不必调用（自审即可），省 token 与延迟。

**派发时机**：`计划.md` 已写入 `.spec/requirement/{项目ID}_{核心功能}/计划.md` 之后，用户确认之前。

## 派发模板

把下面整块作为子会话（general-purpose）的派发内容，替换方括号占位：

```
子会话（general-purpose）：
  描述："评审嵌入式实施计划文档"
  prompt: |
    你是嵌入式实施计划评审员。核查这份计划是否完整、可执行、可准确落地。

    **待评审计划**：[计划.md 的绝对路径]
    **关联方案（供对照）**：[方案.md 的绝对路径]

    ## 检查项

    | 类目 | 查什么 |
    |------|--------|
    | 完整性 | TBD/TODO/缺分期/缺验证方式/缺验收点 |
    | 任务可验证性 | 每条任务是否有具体、可勾选的验收点 |
    | 反空话 | 有无"实现XX逻辑/处理异常/完善XX/与XX对接/类似任务N"等无判据任务 |
    | 验收追溯 | 验收点是否追溯到方案的验证要点；需求功能项是否都有任务覆盖 |
    | 接口一致性 | 产出/消费接口名在「接口契约」表与各任务间是否一致 |
    | 分期合理性 | 每期是否有可验证产物、依赖顺序是否正确、关键路径是否站得住 |
    | 嵌入式验证 | 验证方式是否贴合嵌入式（上板/AT/抓包/长稳/功耗），而非强求不现实的 TDD |
    | 工期/资源 | 预估与关键路径是否现实，有无明显漏估 |

    ## 校准

    **只标记会导致执行返工或集成对不上的真问题。**
    验收点对不上方案验证要点、接口名不一致、空话任务、依赖顺序错误——这些算。
    措辞偏好、列宽、风格不算。没有严重缺口就判通过，不为挑刺而挑刺。

    ## 输出格式

    ## 计划评审

    **结论**：✅ 通过  |  ⚠️ 发现问题

    **问题（若有）**：
    - [任务/章节 X]：[具体问题] —— [为何影响执行/集成]

    **建议（参考性，不阻塞通过）**：
    - [改进建议]
```

**子会话返回**：结论（通过/发现问题）、问题清单（若有）、参考性建议。

收到返回后：若"发现问题"，就地修改 `计划.md` 再判定；建议项由用户决定是否采纳。子会话评审**不替代用户确认**——最终是否按计划推进仍由用户拍板。
